# scenario_timer — 情境節奏計時器

> v1 範圍鎖定。動工前讀這份。

## 定位
不是「鬧鐘」,是**已承諾活動的節奏控制器**。第一個情境:健身組間休息。
接在既有 `features/alarm_engine` 的聲音/震動層上,但**不走背景排程**——
它是**前景、即時、互動的狀態機**(使用者練的時候 App 開著、人盯著組間)。

## 互動模型(Model C — LOCKED)
休息「正數」計時,使用者準備好下一組時自己按。計時器**服務**你的準備,而不是命令你。

```
開始前:設定 目標組數(如 5)+ 軟目標休息(如 90s)

每組迴圈:
  做完一組 ──▶ 點全螢幕大按鈕 ──▶ 休息正數計時 ↑
                                      ├─ 到 90s(軟目標) → 輕震「建議開始」
                                      └─ 到 135s(脊椎)  → 較強震動(不強制)
  準備好 ──▶ 點大按鈕 ──▶ 記錄真實休息秒數 → 組數+1
  做完最後一組 ──▶ 總結畫面(每組真實休息 + 總時間)
```

## 狀態機
| Phase | 意義 | 大按鈕文字 |
|-------|------|-----------|
| setup | 設定組數 + 軟目標 | 開始 |
| working | 正在做這組(不計時),等使用者按 | 做完這組 |
| resting | 休息正數計時中,軟/脊椎提示在此觸發 | 我好了 |
| summary | 達目標組數,顯示每組真實休息 | 完成 |

轉移:`setup→working→(做完這組,組數++)→resting→(我好了,記錄休息)→working→…`
最後一組做完 → 直接 `summary`(最後一組後不需休息)。

## 鎖定的設計決定
- **彈性紀律**:軟提示(輕震)+ 超過軟目標 50% 的脊椎(較強震),都不強制停止。
- **收尾**:先設目標組數,做完跳總結,自動蒐集每組**真實休息時間**(固定計時器拿不到的數據)。
- **介面**:一顆全螢幕、免看螢幕的大按鈕,狀態切換「做完這組 ↔ 我好了」。
- **預設**:軟目標 90s(可調,單一全域值);脊椎 = 軟目標 ×1.5;只計休息、不計做組。

## v1 刻意不做
語音、雲端同步、多情境、每組不同目標、計算做組時間。全部 v2+。

## UI
**全新螢幕,不沿用水滴鬧鐘 mockup**(那是鬧鐘畫面,需求完全不同)。
視覺設計一律先過 UIUX agent 再實作。

## 重用 vs 不重用 alarm_engine
- ✅ 重用:`NotificationScheduler` / `LocalNotificationImpl`(息屏兜底叮震,軌道 B)、Drift 資料層、`@riverpod` Notifier 慣例。
- ❌ 不用:`reschedule_window`(背景精確鬧鐘配額機制,對前景計時器過度工程)、`rrule_engine`、`minuteTickProvider`(太粗,另做秒級 ticker)。

## 提示輸出/輸入(2026-06-30 耳機研究結論)
- ✅ **私密耳機叮聲 = 一等提示**:叮聲蓋過音樂(ducking),手機放遠也聽得到、不吵別人。
  依賴 `audio_session`(duckOthers)+ `just_audio`;iOS 叮完要主動 deactivate 還原音量。
- ❌ **耳機手勢輸入「進下一組」= 放棄**:iOS 無合法路;Android 僅 AccessibilityService 髒路且很可能收不到/順帶暫停音樂。連 spike 都省。
- 🟡 **免手機輸入 = 待驗證**:醜版先用「手機輕觸」;頭 3 次健身驗證是否真需要免手機輸入,需要再上藍牙快門環(候選)。
